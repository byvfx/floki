# Floki Codebase Audit — June 2026

Full-codebase health review focused on **maintainability**, **pragmatism**, and
**performance**. floki is a GPU-accelerated multi-layer OpenEXR viewer (egui + wgpu +
the `exr` crate, with an optional feature-gated OCIO C++ sub-crate).

Findings are prioritized with `file:line` references. Each actionable item is tracked as a
GitHub issue (linked below); this document is the single source those issues reference.

| [#28](https://github.com/byvfx/floki/issues/28) | perf: async EXR decode | P1 |
| [#26](https://github.com/byvfx/floki/issues/26) | maint: break up `ExrViewer::ui()` | M1 |

| [#31](https://github.com/byvfx/floki/issues/31) | maint: low-priority cleanups | T1–T3 |

**Headline: the codebase is in good shape.** Strict CI (`clippy -D warnings`, `fmt`,
multi-platform build matrix), 61 tests, `thiserror` for errors, FFI `unsafe` cleanly
isolated to the cxx bridge, good GPU caching (display transform cached + scissored as of
#23). There is no rescue work here — what follows is a short list of targeted improvements,
roughly highest-payoff first.

## Tier 1 — Maintainability (highest payoff)

### M1. `ExrViewer::ui()` is a ~1244-line method — `src/viewer.rs:579` ([#26](https://github.com/byvfx/floki/issues/26))
The single biggest maintainability risk. One method owns: OCIO cache invalidation, blink
animation, control rows, texture-cache resizing, contact-sheet layout (a large nested
closure at `:639`), per-layer texture generation, all six compare modes, the paint
callback wiring, hover/sample readout, and the status bar. Deep closure nesting compounds it.

**Recommendation (extraction only — no behavior change):**
- `draw_contact_sheet(...)` — lift the `draw_sheet` closure (`:639–738`) to a method.
- `sync_texture_caches(layer_count, layer_count_b)` — the resize blocks (`:624–636`).
- `invalidate_ocio_cpu_textures(...)` — the `#[cfg(feature="ocio")]` block (`:591–606`).
- `draw_canvas(...)` / `emit_paint_callback(...)` — the main render + callback path.
- Keep `ui()` as a thin orchestrator that calls these in order.
- Guardrail: mechanical. The existing `egui_kittest` harness + `cargo test --all-targets`
  should pass unchanged; do it in small commits.

### M2. Sparse public-API docs in the binary crate — `src/app.rs`, `src/viewer.rs` ([#29](https://github.com/byvfx/floki/issues/29))
`floki-ocio` is well-documented (~30 `///`), but the main crate's public types and methods
mostly lack doc comments (`ExrApp`, `ExrViewer`, the `generate_*_texture` family,
compare-mode helpers). Internal `//` comments are good where they exist.
**Recommendation:** add `///` headers to public items and to the non-obvious
`generate_texture` vs `generate_texture_ocio` vs `generate_composite_texture_ocio` split,
explaining when each path is taken. Cheap, high readability payoff.

### M3. Document the texture-generation family's contract — `src/viewer.rs:1824+`, `:2061+`, `:2409+` ([#30](https://github.com/byvfx/floki/issues/30))
Five sibling functions (`generate_texture`, `generate_texture_ocio`,
`generate_diff_texture`, `generate_composite_texture`, `generate_composite_texture_ocio`)
share structure but differ by blend/transform path. The duplication is *justified*, not a
defect — but the selection logic is implicit. **Recommendation:** a short module-level doc
block (a small table: mode × OCIO on/off → which fn, which cache key) so the next reader
doesn't have to reverse-engineer it. Do **not** force a premature merge.

## Tier 2 — Performance (user-visible, smaller surface)

### P1. Image decode is synchronous on the UI thread — `src/app.rs:356` → `src/exr_loader.rs:37` ([#28](https://github.com/byvfx/floki/issues/28))
`open_file()` calls `ExrData::load()` directly from menu click handlers; the `exr` read +
miniz_oxide decompression blocks the UI thread. Large multi-layer EXRs visibly freeze the
window during open. (`catch_unwind` at `exr_loader.rs:46` is defensive only.)
**Recommendation:** decode on a worker thread (the converter tool already uses
`std::thread::spawn` at `app.rs:499` as a pattern), hand the result back via channel, show
a loading state, and `request_repaint()` on arrival. Biggest perceived-responsiveness win.

### P2. OCIO display-transform caching — already done, just confirm — `src/gpu/ocio_pass.rs:450`
The FNV-1a `render_sig` cache (skip both passes on clean repaint) + scissor to visible
bounds (`:509`) + `Rgba16Float` offscreen are all in place and correct. **No action**;
noted so it isn't "re-optimized." Worth a regression test if the signature inputs ever change.

### P3. CPU diff/composite buffer allocations — acceptable, watch only — `src/viewer.rs:2185, 2280, 2452, 2522`
Full `vec![...; w*h]` buffers per computation, but correctly cached on param change and
parallelized via rayon — only runs on mode/param change, not per frame. **No action** for a
pro viewer; revisit only if interactive scrubbing of diff multiplier ever feels laggy
(then reuse a persistent buffer).

## Tier 3 — Nice-to-have / low priority ([#31](https://github.com/byvfx/floki/issues/31))

- **T1. Test coverage of render paths** — `viewer.rs` rendering is largely untested
  (`egui_kittest` is available as a dev-dep but underused). Low priority given GPU testing
  cost; consider a smoke test that drives `ui()` headless once compare modes stabilize.
- **T2. `CubeLut` dead-code flag** — `src/color/cube.rs:5` `#[allow(dead_code)]`: fully
  parsed/tested but not wired into the UI. Either wire it up or add a one-line note on the
  allow explaining it's staged for a future feature, so it doesn't read as rot.
- **T3. Explicit fallbacks over silent defaults** — `floki-ocio/src/backend.rs:109,171,178`
  use `unwrap_or(0.0)` on iterators. Safe today; a comment on *why* 0.0 is correct would
  prevent a future misread.

- Error handling: `Result`-based, `thiserror` in `floki-ocio/src/error.rs`, no production
  panics; the few `unwrap()`s are guarded (`viewer.rs:1243`, `:992`; egui `get::<T>()`
  contract in `ocio_pass.rs`).
- `unsafe`: confined to the cxx FFI bridge (`floki-ocio/src/ffi.rs:49`) + one test env var.
- Build/CI: `.github/workflows/build.yml` enforces fmt + `clippy -D warnings` + all-targets
  tests across Linux/macOS/Windows, with feature-matrix (non-OCIO + vendored-OCIO).
- Module boundaries, naming consistency, no commented-out code or unused imports.

## Performance Testing Harness (to be implemented — [#27](https://github.com/byvfx/floki/issues/27))

A standing benchmark suite so perf is tracked, not guessed. **Decision:** `criterion`
dev-dependency + `benches/`; real-world EXRs live in a **gitignored local dir** (no repo
bloat), CI runs **synthetic** fixtures only.

### Tooling
- Add `criterion` under `[dev-dependencies]` and a `[[bench]]` entry (`harness = false`) in
  `Cargo.toml`. Statistical timing + regression detection + HTML reports in `target/criterion/`.

### What to benchmark (deterministic CPU/decode paths)
- `ExrData::load()` — decode + channel→logical-layer grouping (`src/exr_loader.rs:37`). The
  P1 hotspot; the headline number.
- `longest_common_dotted_prefix` / layer grouping on wide Blender-style channel sets
  (100+ channels) — `src/exr_loader.rs`.
- Histogram build, log + linear (`src/viewer.rs:2643+`).
- CPU diff (`src/viewer.rs:2171+`) and composite (`:2250+`) over a full-res buffer.
- GPU frame-time is **out of scope** — CI can't measure it deterministically; keep it
  manual / optionally instrumented behind a log flag.

### Two-tier fixtures
- **CI tier — synthetic, in-repo-free:** generate large multi-layer EXRs *in-code* at bench
  setup (reuse the EXR-writing pattern already in the `exr_loader` tests). Reproducible,
  runs anywhere, catches regressions on every push. Vary: resolution, layer/channel count,
  compression (none / ZIP / PIZ).
- **Local tier — real renders:** resolve from a gitignored `assets/perf/` dir (or a
  `FLOKI_PERF_FIXTURES` env var). Benches **skip gracefully with a log line** when the dir
  is absent, so the suite still runs in CI and for fresh clones. Add `assets/perf/` to
  `.gitignore` and document the expected layout in a short `assets/perf/README.md`.

### Verification
- `cargo bench` runs clean with no local fixtures present (synthetic-only path), and runs
  the extra real-render cases when `assets/perf/` is populated.
- Numbers are stable across runs (criterion variance reported); record a baseline so future
  changes (esp. P1 async decode) can be compared against it.
